# 구독청소 작업계획서

작성 기준: 2026-06-29

## 1. 제품 정의

**구독청소**는 사용자가 놓치기 쉬운 자동결제, 구독, 멤버십, 렌탈, 정기 후원, 앱스토어 결제를 찾아서 정리하도록 돕는 생활비 관리 도구다.

핵심 포지션은 가계부가 아니라 **새는 돈 점검 도구**다. 모든 지출을 기록하게 만들지 않고, 반복해서 빠져나가는 돈만 빠르게 보여준다.

## 2. 첫 사용자

초기 타깃은 다음 세 그룹이다.

- OTT, 쇼핑 멤버십, AI 툴, 클라우드, 앱스토어 구독을 여러 개 쓰는 20~40대
- 자동결제는 많은데 카드/계좌 앱을 매번 확인하기 귀찮은 직장인
- 부모님, 가족, 배우자의 반복결제를 대신 정리해주고 싶은 사용자

## 3. 핵심 문제

어카운트인포는 자동이체/자동납부 확인에 강하지만, 실제 생활 속 자동결제는 카드 승인 내역, 앱스토어 결제, 간편결제, 통신 부가서비스, 해외 구독, 렌탈, 멤버십으로 흩어져 있다.

구독청소는 이 흩어진 결제 흔적을 한 번에 붙여 넣고 다음 질문에 답하게 한다.

- 이번 달 반복해서 빠지는 돈은 얼마인가?
- 내가 기억 못 하는 결제처가 있는가?
- 다음 결제 전 확인해야 할 항목은 무엇인가?
- 유지할 것과 해지할 것을 어떻게 나눌 것인가?

## 4. 제품 구조

### 정보 구조

- 결제내역 입력
- 반복결제 후보
- 월 예상 지출
- 월 절약 목표
- 다음 결제 예상일
- 상태 관리: 유지, 해지예정, 모름
- 원문 거래 목록

### 데이터 모델

`Transaction`

- `id`: 거래 식별자
- `date`: 결제일
- `merchant`: 결제처
- `amount`: 금액
- `currency`: 통화
- `cardProvider`: 카드사/간편결제사. 예: 신한카드, 현대카드, 카드 미확인
- `raw`: 원문

`SubscriptionCandidate`

- `key`: 정규화된 결제처 키
- `merchant`: 표시용 결제처
- `category`: 분류
- `cardProviders`: 후보에 연결된 카드사 목록
- `occurrences`: 반복 횟수
- `averageAmount`: 평균 금액
- `cadence`: 주기
- `nextDate`: 다음 결제 예상일
- `confidence`: 신뢰도
- `status`: 유지, 해지예정, 모름

## 5. 단계별 진행 계획

### 1단계: API 없는 검증형 모바일 웹

목표: 개인정보와 심사 부담 없이 "사람들이 이 리포트를 실제로 계속 쓰는지" 검증한다.

포함 기능:

- 카드 승인 문자, 앱 푸시 알림, 카드 명세서 텍스트 복붙
- 날짜/금액/결제처 자동 추출
- 동일 결제처 반복 탐지
- 월 예상 지출 계산
- 다음 결제 예상일 표시
- 유지/해지예정/모름 상태 저장
- 해지예정 항목 기준 월 절약 목표 진행률
- 브라우저 로컬 저장
- 모바일 우선 화면

제외 기능:

- 카드/은행 자동 연동
- OCR
- 로그인
- 서버 저장
- 문자/카카오 알림 발송
- 결제 유료화

성공 기준:

- 사용자가 3분 안에 자신의 반복결제 후보를 확인한다.
- 결제내역 20줄 기준 반복결제 후보가 눈에 띄게 정리된다.
- 사용자가 최소 1개 항목에 `해지예정` 상태를 남긴다.

### 2단계: API 필요 기능 추가

목표: 분석 정확도와 편의성을 높인다.

필요 API 후보:

- OpenAI API: 결제처 의미 분석, 카테고리 분류, 해지 방법 요약
- OCR API: 카드 명세서 캡처/이미지에서 거래 내역 추출
- Supabase/Firebase: 로그인, 사용자별 데이터 저장
- 이메일/SMS/알림톡 API: 결제 전 리마인드

포함 기능:

- 결제처 이름 자동 정리
- 해외 구독 원화 환산 보조
- 해지 링크/해지 경로 안내
- 월간 절약 리포트
- 가족 공유용 읽기 링크
- 고객문의/방명록형 피드백 수집

주의:

- 의료/금융 판단처럼 오해될 표현을 피한다.
- "해지 대행"보다 "해지 경로 안내"로 시작한다.
- 민감 정보는 최소 수집, 최소 저장한다.
- 고객문의에는 카드번호, 주민등록번호, 인증서, 비밀번호, 계좌번호 전체를 받지 않는다.

### 2.5단계: 고객문의/방명록

목표: 자동 금융 연동 전에 실제 사용자 불편과 요청을 모은다.

초기 형태:

- 이름 또는 닉네임
- 이메일 또는 연락처 선택 입력
- 문의 유형: 분석 오류, 결제처 추가 요청, 해지 방법 요청, 기능 제안
- 문의 내용
- 선택 동의: 답변을 위한 연락 동의

운영 기준:

- 처음에는 Supabase/Firebase 같은 간단한 DB 또는 Google Form/Notion Form으로 시작할 수 있다.
- 문의 작성 화면에는 민감정보를 쓰지 말라는 안내를 둔다.
- 접수된 문의를 보고 카드사별 문자 패턴, 자주 나오는 결제처, 필요한 해지 가이드를 우선순위화한다.
- 문의 기능은 제품 검증용이며, 금융 상담이나 해지 대행으로 표현하지 않는다.

### 3단계: 앱 단계

목표: 반복 사용과 알림을 강화한다.

진행 순서:

1. PWA 완성: 홈 화면 추가, 오프라인 캐시, 반응형 UI 안정화
2. 앱 래핑 검토: Capacitor 또는 React Native로 앱스토어 진입
3. 네이티브 알림: 결제 예정 푸시, 해지예정 리마인드
4. 파일/이미지 가져오기: 명세서 캡처, PDF, CSV 업로드
5. 보안 강화: 기기 내 잠금, 민감 문구 마스킹, 데이터 내보내기/삭제

앱에서 조심할 점:

- SMS 자동 읽기는 권한, 심사, 개인정보 부담이 크므로 후순위다.
- 초반에는 사용자가 직접 붙여 넣거나 파일을 선택하는 방식이 안전하다.
- 카드/은행 자동 연동은 마이데이터/오픈뱅킹 요건 때문에 별도 사업 단계로 본다.

### 3.5단계: 본인인증/금융 연동 검토

목표: 사용자가 직접 붙여 넣지 않아도 결제 내역을 가져올 수 있는지 검토한다.

후보:

- Toss 인증/간편인증
- PASS/휴대폰 본인확인
- 모바일 신분증 인증
- 공동인증서/금융인증서 기반 인증
- 마이데이터 API
- 오픈뱅킹 API
- 카드사/PG사 제휴 API

진입 조건:

- 사용자 문의와 베타 사용에서 자동 연동 수요가 명확하다.
- 사업자, 개인정보 처리방침, 이용약관, 보안 정책을 준비한다.
- 금융/본인인증 제휴 가능성과 비용을 확인한다.
- 민감정보 저장 범위, 암호화, 접근 권한, 삭제 요청 프로세스를 설계한다.

주의:

- 공동인증서나 모바일 신분증 인증은 단순 API 키 추가 수준이 아니다.
- 마이데이터/오픈뱅킹은 심사, 보안, 사업 요건이 있어 MVP 초반 기능으로 넣지 않는다.
- 자동 연동 전까지는 `붙여넣기/CSV/OCR` 기반으로 제품 가치를 먼저 검증한다.

### 4단계: 수익화 단계

웹 수익화:

- 무료: 최근 3개월 텍스트 분석, 후보 개수 제한, 로컬 저장
- 유료: 12개월 분석, 고급 분류, 리포트 저장, 가족 공유, 결제 전 알림
- B2B: 프리랜서/소상공인 SaaS 비용 점검 리포트

앱 수익화:

- 월 구독: 2,900원~4,900원
- 연 구독: 월 환산 할인
- 가족 플랜: 부모님/배우자/자녀 결제 관리
- 프리미엄 알림: 결제 7일/3일/1일 전 푸시

추가 수익화 후보:

- 해지 가이드 프리미엄
- 지출 절감 챌린지
- 소상공인용 SaaS 비용 진단
- 카드/통신/보험 비교 제휴는 신뢰가 쌓인 뒤 신중히 검토

## 6. 초기 기술 선택

1단계는 정적 웹으로 시작한다.

- `index.html`
- `styles.css`
- `app.js`
- `localStorage`
- PWA manifest
- 선택적 service worker

이 선택의 장점:

- API 키가 필요 없다.
- 서버 비용이 없다.
- 개인정보 저장 부담이 낮다.
- 모바일 웹 링크로 바로 테스트할 수 있다.
- 나중에 React/Next.js 또는 앱으로 옮기기 쉽다.

## 7. 1단계 테스트 시나리오

테스트 사용자는 최근 2~3개월 카드 결제 알림을 10~30줄 붙여 넣는다.

확인할 질문:

- 반복결제 후보가 납득 가능한가?
- "이 결제 뭐지?" 싶은 항목이 보이는가?
- 유지/해지예정/모름 분류가 자연스러운가?
- 다음 결제 예상일이 유용한가?
- 다시 열었을 때 저장되어 있는 게 편한가?
